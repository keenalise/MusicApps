## How to Use in Your App

### Step 1: Add Sessions Tab to Navigation
In MainActivity.kt, add the sessions screen to your NavHost:

```kotlin
composable(Screen.Sessions.route) {
    val sessionViewModel: SessionManagementViewModel = viewModel()
    SessionsScreen(
        viewModel = sessionViewModel,
        onSessionSelected = { sessionId ->
            // Optional: navigate to player or handle selection
        }
    )
}
```

### Step 2: Create a Session from UI
Users can tap the + button in SessionsScreen to create an empty session.

### Step 3: Add Songs to Session
Option A - From CreateSessionFromLibraryScreen:
```kotlin
// Show screen for user to select songs
CreateSessionFromLibraryScreen(
    musicViewModel = musicViewModel,
    sessionViewModel = sessionViewModel,
    onSessionCreated = { newSessionId ->
        // Session created with selected songs and auto-playing
    }
)
```

Option B - Programmatically:
```kotlin
val sessionQueueManager = SessionQueueManager(sessionRepository)
sessionQueueManager.replaceSessionQueue(sessionId, listOfSongs)
```

### Step 4: Switch Sessions
```kotlin
// From any ViewModel or Composable
sessionViewModel.switchToSession(
    sessionId = 123,
    autoPlay = true  // Optional: auto-start playing
)

// What happens:
// 1. Current session state saved (position, index, state)
// 2. New session queue loaded into ExoPlayer
// 3. Player seeks to saved position
// 4. Playback starts if autoPlay=true
```

## Key Classes Explained

### SessionManagementViewModel
- Main entry point for UI
- Manages session CRUD (Create, Read, Update, Delete)
- Integrates with SessionManager for playback control
- Exposes: allSessions (StateFlow), activeSession (StateFlow)

### SessionManager
- Handles atomic session switching
- Synchronizes ExoPlayer with session state
- Automatically saves playback position before switching
- Restores exact position when returning to session

### SessionRepositoryImpl
- Persists sessions to Room database
- Manages Flow-based observation
- Handles session queries and mutations

### SessionQueueManager
- Converts Song objects to QueueItems for persistence
- Validates URIs (security: only content://, http://, https://)
- Bridges music library with session system

## Complete Usage Example

```kotlin
// In your Composable or ViewModel

// 1. Get ViewModels
val musicViewModel: MusicViewModel by viewModel()
val sessionViewModel: SessionManagementViewModel by viewModel()

// 2. Load device music
LaunchedEffect(Unit) {
    musicViewModel.loadSongs()
}

// 3. Observe all sessions
val allSessions by sessionViewModel.allSessions.collectAsStateWithLifecycle()

// 4. Create new session with songs
val allSongs by musicViewModel.allSongs.collectAsStateWithLifecycle()

LaunchedEffect(allSongs) {
    if (allSongs.isNotEmpty()) {
        // Create "Workout" session with first 20 songs
        val workoutSessionId = sessionViewModel.sessionRepository.createSession(
            "Workout",
            allSongs.take(20).map { song ->
                com.example.musicc.domain.QueueItem(
                    mediaId = song.id,
                    uri = song.uri?.toString() ?: "",
                    title = song.title,
                    artist = song.artist,
                    album = song.album,
                    durationMs = song.durationMs
                )
            }
        )
    }
}

// 5. Switch to session
Button(onClick = {
    sessionViewModel.switchToSession(workoutSessionId, autoPlay = true)
}) {
    Text("Play Workout Mix")
}
```

## Database Schema

```
playback_sessions table:
- id (PRIMARY KEY)
- title (session name)
- created_at (timestamp)
- updated_at (timestamp)
- current_index (current track in queue)
- last_position_ms (playback position)
- playback_state (0=STOPPED, 1=PAUSED, 2=PLAYING)
- repeat_mode (repeat setting)
- shuffle_mode_enabled (boolean)
- is_active (current active session flag)

queue_items table:
- id (PRIMARY KEY)
- session_id (FOREIGN KEY → playback_sessions.id)
- media_id (song identifier)
- uri (file path/content URI)
- title, artist, album (metadata)
- duration_ms (song length)
- position_in_queue (order in playlist)
- extras_json (reserved for future)
```

## State Flow Diagram

```
User taps "Switch Session"
        ↓
SessionManagementViewModel.switchToSession()
        ↓
SessionManager.switchToSession()
        ↓
1. Save current session state to DB
        ↓
2. Load new session from DB
        ↓
3. Replace ExoPlayer queue with new session's songs
        ↓
4. Seek to saved position
        ↓
5. Start/pause based on autoPlay flag
        ↓
6. Mark new session as active in DB
        ↓
UI updates to show new active session
```

## Security Features Implemented

✓ **Path Traversal Prevention**: All URIs validated against whitelist
  - Only content:// (local files), http://, https:// allowed

✓ **Memory Safe**: Single ExoPlayer instance
  - No leaks from multiple player instances
  - Sessions switch by queue mutation, not player recreation

✓ **Database Constraints**: Foreign key cascade
  - Deleting session auto-deletes its queue items
  - Data integrity guaranteed

✓ **URI Validation**: Before any playback
  - Malformed URIs caught and handled
  - Invalid schemes rejected early

## Common Issues & Solutions

### "No sessions showing"
- Check: SessionsScreen navigation added
- Check: Permissions granted
- Check: Room database initialized
- Try: Rebuild project (./gradlew clean build)

### "Sessions don't switch"
- Check: ExoPlayer initialized
- Check: Songs have valid URIs
- Check: SessionManager has player reference
- Try: Restart app

### "Position not restoring"
- Check: saveActiveSessionState() called
- Check: Player.prepare() after setMediaItems()
- Check: currentIndex correct in queue
- Try: Check logcat for errors

### "App crashes on session switch"
- Check: All songs have valid URIs
- Check: ExoPlayer not released while in use
- Check: Coroutine scope still active
- Try: Add error handling in catch blocks

## Performance Tips

1. **Large Libraries** (1000+ songs)
   - Don't load all songs to every session
   - Use pagination when selecting songs
   - Cache session list with Flow.stateIn()

2. **Many Sessions** (50+)
   - Archive old sessions
   - Lazy-load queue items per session
   - Implement session search

3. **Long Queues** (500+ songs per session)
   - Lazy-load queue items
   - Batch database inserts
   - Consider pagination in player UI

4. **Memory**
   - Use single ExoPlayer (already done)
   - Release sessions on destroy
   - Clear error state periodically
