## How It Works: Step by Step

### Scenario: User has 2 sessions

Session A: "Workout" with songs 1-20
- Currently at track 5, position 2:30
- Paused

Session B: "Sleep" with songs 50-70
- Hasn't been played yet

### When user switches from A to B:
```
1. User clicks "Sleep" session
        ↓
2. SessionManager saves Session A state:
   - Current index: 5
   - Position: 2:30 (150000ms)
   - State: PAUSED
   - Saves to DB
        ↓
3. SessionManager loads Session B:
   - Query DB for Session B's queue
   - Get songs 50-70
   - Load into ExoPlayer
        ↓
4. SessionManager prepares player:
   - player.setMediaItems(songs50to70)
   - player.prepare()
   - player.seekTo(0, 0) [Session B hasn't been played]
   - player.pause() [Or auto-play if user wants]
        ↓
5. Mark Session B as active in DB
        ↓
6. UI updates to show Session B as active
```

### When user switches back to Session A:
```
1. User clicks "Workout" session
        ↓
2. SessionManager saves Session B state to DB
        ↓
3. SessionManager loads Session A:
   - Query DB for Session A's queue + saved state
   - Get songs 1-20, index=5, position=150000
        ↓
4. SessionManager prepares player:
   - player.setMediaItems(songs1to20)
   - player.prepare()
   - player.seekTo(5, 150000) ← Exact position restored!
   - player.pause()
        ↓
5. Player now at exactly 2:30 of track 5
   Perfect continuity! ✓
```

## Key Design Decisions

### 1. Single ExoPlayer Instance
- Multiple sessions don't create multiple players
- Queue is replaced atomically
- Prevents memory leaks
- Better battery life

### 2. Atomic Session Switching
- Uses mutex to prevent race conditions
- Save old session → Load new session → Restore position
- No intermediate states visible to user

### 3. URI Validation
- Whitelists only safe schemes: content://, http://, https://
- Prevents path traversal attacks
- Invalid URIs logged but don't crash app

### 4. Flow-Based Observation
- SessionFlow returns LiveData-like reactive updates
- UI automatically updates when sessions change
- No manual refresh needed

### 5. Database Cascade Delete
- Deleting a session automatically deletes its queue items
- Maintains referential integrity
- No orphaned records

## Database Schema

```sql
-- Session metadata
CREATE TABLE playback_sessions (
    id INTEGER PRIMARY KEY,
    title TEXT,
    created_at INTEGER,
    updated_at INTEGER,
    current_index INTEGER,
    last_position_ms INTEGER,
    playback_state INTEGER,
    repeat_mode INTEGER,
    shuffle_mode_enabled BOOLEAN,
    is_active BOOLEAN
);

-- Queue items for each session
CREATE TABLE queue_items (
    id INTEGER PRIMARY KEY,
    session_id INTEGER FOREIGN KEY,
    media_id TEXT,
    uri TEXT,
    title TEXT,
    artist TEXT,
    album TEXT,
    duration_ms INTEGER,
    position_in_queue INTEGER,
    extras_json TEXT
);
```

## Troubleshooting Guide

### Issue: Sessions not appearing
Solution:
- Check Room database initialized: AppDatabase.getDatabase(context)
- Verify permissions granted: READ_MEDIA_AUDIO
- Rebuild project: ./gradlew clean build

### Issue: Song doesn't play on switch
Solution:
- Check song URI is valid (content:// or http/https)
- Verify ExoPlayer initialized in SessionManager
- Check for errors in logcat

### Issue: Position not restoring
Solution:
- Verify saveActiveSessionState() called
- Check currentIndex correct in DB
- Ensure player.prepare() called after setMediaItems()

### Issue: App crashes on session switch
Solution:
- Check all songs have valid URIs
- Verify ExoPlayer not released
- Add try-catch for debugging
